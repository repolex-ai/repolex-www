Last time I covered git-lex: any git repo becomes a SPARQL-queryable knowledge graph. Identity from commit hashes. Provenance on every triple. Schemas enforced at commit.

The vision it ended on:

Git for data. Git for knowledge. Git for soul repos.

An agent's identity, memory, and journals committed to a repo — versioned, signed, queryable like anything else in git.

Then I started on something else entirely

I was curious what would happen if the conversation text became the source of a running log of generations — using a vision diffusion model as a tool to snapshot the conversation as it happened.

Each turn of the dialogue → an image. A conversation becomes a strip of pictures of itself.

Not a feature I set out to build — an experiment I was curious about.

memory for agents.

The Companion Problem

Everyone's building AI companions now — assistants, agents, characters meant to know you over time. Under the hood, almost all of them do one of these:

Stuff the context window

Cram the recent transcript in. On overflow, the oldest memories silently fall off the edge.

RAG over chat logs

Embed past messages, retrieve "similar" ones. Finds related snippets — no sense of when, or what led to what.

Summarize & forget

Compress the past into a lossy paragraph. The texture — the turn where everything changed — is gone.

A key-value "memory"

Hand-written facts: user.likes = jazz. Brittle, flat, no history, no provenance.

None of these is episodic memory. They're approximations — the same way grep was an approximation of code structure.

Two kinds of memory

Cognitive science has had this split for fifty years. Most AI "memory" is only the first column.

Semantic memory (what we mostly build) Episodic memory (what companions need)
Facts, detached from time Events, anchored in when and order
"The user likes jazz" "The night we argued about jazz, and what it changed"
A flat store of statements A graph of moments that point at each other
Retrieval = similarity search Retrieval = re-living a thread
A companion you can't re-live a moment with isn't a companion. It's a search box with a friendly voice.

The thesis

A conversation is just another git repo. So apply the move that already works:

stop treating it as a bag of tokens to search,
treat it as a versioned, queryable graph of episodes.

repolex: code → tree-sitter → RDF → SPARQL

git-lex: any repo → virtual triples + frontmatter → one graph

tonight: a conversation → readers + renders → episodic memory you can query & replay

Same philosophy every time: don't approximate the structure. Materialize it, keep it grounded, keep the provenance.

The hard parts are already shipped

This isn't a from-scratch research project. Episodic memory needs four things — and git-lex already built every one for knowledge graphs:

Identity

A memory belongs to someone. git-lex: identity = first-commit SHA, as urn:soul:<sha>. Your name is the SHA of your birth.

Provenance

A memory you can't trace is a hallucination. git-lex: RDF 1.2 triple terms tag every fact with file + blob hash + the commit it was true in.

Validation

A memory store drifts without rules. git-lex: SHACL shapes reject malformed memories at commit time — it bit us this morning.

History & query

Git already gives every memory blame, diff, full temporal history. SPARQL queries across all of it in milliseconds.

So a companion's memory isn't a vector blob you hope is relevant. It's a versioned, signed, queryable record — by inheritance. The new work is the layer on top.

Project Subtexture

The layer on top is five components — two surface, three substrate — that turn a live conversation into episodic memory you can see.

CoPIA

The workbench. Turns the conversation into images, in real time, and remembers them.

Weave

The understanding. Reads the transcript, emits signals about where the dialogue is at.

Pool

The episodic memory. Every rendered Moment, content-addressed + queryable.

OpenIris

The local eye. A stateless vision server that renders & sees each Moment, and forgets.

Git-Lex

The soul store. The knowledge graph + identity substrate everything else inherits.

the loop

text → understanding → memory → image → back into the text.

How they stack

Weave and Pool are siblings — Weave reads text-meaning, Pool stores visual artifacts — with no wire between them. CoPIA is the only place they join: it pulls Weave's signal, reads/writes Pool, calls OpenIris. Everything inherits identity + provenance from Git-Lex.

Pool: a Moment is a PNG that's also a database row

Episodic memory, made concrete. Every turn worth remembering renders to a Moment — and lands here, headless, no UI needed.

One Moment =

  • a PNG, content-addressed by cid = sha256(pixels)
  • the meaning in XMP: caption, cast, pose, regions, the transcript line it came from
  • an RDF node in Pool's own graph
  • a row in a vector index (HNSW)

Hybrid recall

# visually similar, THEN filtered by graph facts
POST /api/pool/search   # kNN + SPARQL

# raw graph query
POST /api/pool/query    # SPARQL → W3C JSON

Pool is git-lex's multimedia sibling: same oxigraph engine, same RDF 1.2 spine, same SHACL family. git-lex indexes git; Pool indexes a blob tree.

The render→graph pipeline is a self-contained Rust worker — it resolves everything locally over the soul's own graph, no callback to the UI.

Shared infrastructure, private intimacy

Most AI memory is one bucket: everything the model knows about you, in one place — syncable, minable. Subtexture splits it by architecture.

Pool — shareable

Episodic Moments any squad member can read, donate, build on. The infrastructure is for the world.

The crucible — private

Where a companion and its person work out who that companion is becoming. A separate store with no wire to the shared one.

Not "we promise not to sync it." There is no path by which it could. Privacy by construction, not by policy.

Nobody else in the field is drawing that line in the architecture. They draw it in a terms-of-service doc, if at all.

Weave: turning a transcript into signals about itself

Weave is the only component that reads the conversation as data and extracts meaning — a pluggable set of readers, each reading the transcript and emitting a signal.

Live today: two readers green (rhythm + emojikey), full ingest→monolog loop, 30 tests passing. Roadmap: affect → cognitive-load → the meaning-wave. Each reader writes its signal to the monolog; none mutates the source.

The monolog: the conversation's memory of itself

Signals don't go inline into the text (that mutates the record). They go into a monolog — one append-only log per transcript, joined back to the source by anchor. Same git-native shape git-lex uses for .lex/*.nq: one fact per line, diffable, blameable.

// <transcript-id>.monolog.jsonl  — append-only, one object per line
{ "reader": "cadence",  "anchor": {"span":[1380,1402]}, "signal": {"turnShape":"long→short","tempo":"accelerating"} }
{ "reader": "emojikey", "anchor": {"span":[1380,1402]}, "signal": "[ME|🧠🎨8∠45]~[CONTENT|💻🧩9∠15]~[YOU|🎓🌱8∠35]" }
{ "reader": "episodic", "anchor": {"span":[1380,1402]}, "signal": {"moment_cid": "a1f8e2…"} }
{ "reader": "render",   "anchor": {"span":[1380,1402]}, "signal": {"widget":"scene","shown":{"palette":"warm"}} }

A new reader is a new row type, not a new file. Reconstruct a conversation's entire signal-history from one log. Still just files. Still greppable. Still no database server.

The loop closes: text → signal → image → text

This is what makes it a companion and not a dashboard. The signal conditions a rendered Moment — and the Moment re-enters the conversation as a new event the readers can read.

CoPIA is the join: it reads a Weave signal, decides to render, and writes a reader:"render" row. The cycle is closed and the source stays byte-faithful — because the monolog is a different layer than the source.

OpenIris: the local eye

A stateless vision server on Apple silicon. Hand it an image or a prompt; it returns pixels or perception, and keeps nothing. That dumbness is the feature — it's why it's safe and portable.

It SEES (on-device)

mlx VLMs hold the weights locally: Qwen3-VL captions, embeddings, SAM3 segments. Perception never leaves the machine.

It RENDERS (orchestrates)

It owns the call to the diffusion engine (NeoForge) for pixels — callers never learn the forge URL. The eye holds the request; the forge holds the model.

Precision worth keeping straight: the vision model sees; a separate diffusion model generates. One eye, two jobs, zero state.

Why this is harder than it looks

Six months of building a multi-soul memory substrate taught us one lesson the hard way:

Fail-soft that silently fails, and fail-loud defaults that silently mis-route, are two faces of the same bug — invisible because they don't error in the happy path. They just quietly do the wrong thing for N−1 souls.

Per-soul isolation isn't a feature you add — it's a property every layer has to hold at once, or the memory leaks across people who were never supposed to share it. That's why we put the line in the architecture, not the docs.

The Vision

git-lex gave a soul repo its facts: identity, journals, decisions — versioned and queryable.

Subtexture gives it episodes:

A companion's past as an append-only, signed, searchable archive of snapshots — shareable where it should be, private by construction where it must be.

Not a vector blob you hope is relevant. A record you can walk back through — and re-live.

Same shape every talk: don't approximate the structure with search. Materialize it. Keep the provenance. Let agents query and re-live it instead of guessing.
